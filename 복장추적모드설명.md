# 복장 추적 및 프롬프트 강화 시스템

## 시스템 목적

캐릭터의 복장을 **연속된 이미지 생성 간에 일관되게 유지**하는 것.
채팅(RP) 진행 중 캐릭터가 옷을 갈아입지 않았는데 프롬프트/모델이 다른 복장을 생성하면, 이를 이전 복장으로 교정.
반대로 채팅에서 실제로 옷을 갈아입었다면, 새 복장을 자연스럽게 반영.

---

## 전체 파이프라인

```
[Sliyu/LB-LUA 등에서 프롬프트 요청]
        │
        ▼
[server.py] positive + chat 수신
        │
        ▼ ── Phase 1: 프롬프트 강화 (이미지 생성 전)
[prompt_enhance_mode.py]
  │  1. [UPSCALE] 섹션에서 char 블럭 분리 (| 기준)
  │  2. 각 캐릭터 블럭에 대해:
  │     - char_name 추출
  │     - outfit_result ← outfit_mode.character_results[name].llm_result
  │     - previous_chat ← outfit_mode 최신 entry의 chat_content
  │     - enhance_outfit_prompt.run(name, char_block, outfit_result, chat, prev_chat)
  │  3. 강화된 블럭으로 프롬프트 교체
  ▼
[batch_mode.py] 강화된 프롬프트로 ComfyUI 이미지 생성
        │
        ▼ ── Phase 2: 복장 추출 (이미지 생성 후)
[outfit_mode.py]
  │  1. 생성된 이미지를 VLM 복장 추출 워크플로우로 분석
  │  2. 캐릭터별로 CharacterOutfitEntry 저장
  │     - outfit_prompt: VLM이 분석한 복장 설명
  │     - positive_prompt: 이미지 생성에 사용된 프롬프트
  │     - chat_content: 당시 채팅 내용
  ▼
[haganai_extract_outfit_prompt.py] (수동 또는 자동 트리거)
  │  1. outfit_list(최근 3개) + chat_list + previous_result 수집
  │  2. LLM이 복장 변화 감지 + 통합 결과 생성
  │  3. JSON 출력 → outfit_mode의 llm_result에 저장
  │     {"headgear":"", "clothing":"", "shoes":"", "accessories":"", "memo":""}
  ▼
  → 이 결과가 다음 배치의 Phase 1에서 outfit_result로 사용됨
```

---

## 핵심 컴포넌트 역할

### 1. outfit_mode.py — 복장 데이터 수집 및 저장
- 배치 완료 후 생성된 이미지를 VLM 워크플로우로 분석
- 캐릭터별로 최대 10개의 CharacterOutfitEntry 보관
- `llm_result`: haganai_extract_outfit_prompt의 JSON 통합 결과
- 디스크에 영속화하여 서버 재시작에도 유지

### 2. haganai_extract_outfit_prompt.py — 복장 통합 및 변화 감지
- **입력**: outfit_list(시간순), chat_list, previous_result(이전 통합 결과)
- **역할**:
  - 이전 복장 상태 vs 새로 들어온 복장 데이터 비교
  - 채팅 맥락으로 복장 변화 감지 (갈아입었는지, TPO 변화 등)
  - 변화 없으면 이전 복장 유지, 변화 있으면 새 복장 반영
  - CHAR_IDENTITIES: 캐릭터별 고유 스타일 정보 (참고용 context)
- **출력**: JSON `{"headgear":"...", "clothing":"...", "shoes":"...", "accessories":"...", "memo":"..."}`
- **중요**: outfit_result가 정확해야 enhance_outfit_prompt도 정확함

### 3. prompt_enhance_mode.py — 강화 오케스트레이션
- 배치 모드의 before_generate 콜백으로 동작
- [UPSCALE] 섹션 파싱, 캐릭터 분리, 강화 스크립트 실행
- 강화 전/후 프롬프트 추적 (재전송 예약용)

### 4. enhance_outfit_prompt.py — 개별 캐릭터 블럭 강화
- **입력**:
  - `char_block`: 현재 프롬프트의 캐릭터 블럭
  - `outfit_result`: haganai의 JSON 통합 결과 (ground truth)
  - `chat_content`: 현재 채팅
  - `previous_chat`: 이전 복장 통합 시점의 채팅
- **핵심 논리 (의사결정 프레임워크)**:
  1. outfit_result를 파싱하여 headgear/clothing/shoes/accessories 파악
  2. char_block의 현재 복장과 비교 → 차이점 식별
  3. chat_content로 판단:
     - **아이템 누락** (outfit_result에 있는데 char_block에 없음) → 채워줌
     - **현재 복장이 틀림** (chat에 변경 언급 없는데 다른 옷) → 이전 복장 유지
     - **이전 예측이 맞지 않음** (chat에 실제 복장 변경 언급) → 현재 복장 사용
  4. 결정된 복장의 디테일을 강화 (소재, 핏, 컬러, 악세서리)
- **주의**: 복장의 **타입/본질**을 바꾸지 않음. "수녀복" → "교복" 금지.

## 데이터 흐름 예시 (takayama maria)

```
[이전 배치 결과]
outfit_result: {
  "headgear": "dark blue cornette, red heart hair ornament",
  "clothing": "dark-blue one-piece cotton nun dress, white collar, white cuffs",
  "shoes": "Unknown",
  "accessories": "None",
  "memo": "NOT CHANGED. ..."
}

[현재 요청]
char_block: "...takayama maria... dark-blue one piece cotton dress, dark blue Cornette"
chat_content: "마리아가 부실에서 장난치고 있음"

→ enhance_outfit_prompt 판단:
  1. outfit_result 파싱: headgear=cornette+빨간하트장식, clothing=수녀복원피스
  2. char_block 비교: cornette는 있지만 빨간하트장식 누락, "nun dress" 표현도 없음
  3. chat 판단: 옷 갈아입었다는 언급 없음 → outfit_result가 정답
  4. 결과: 빨간하트장식 추가, 수녀복 타입 유지, 디테일 강화
```

## 로그 파일

| 로그 | 위치 | 내용 |
|------|------|------|
| 강화 I/O 로그 | `logs/enhance_prompt_io.log` | enhance_outfit_prompt의 **입력 전체**(char_block, outfit_result, previous_enhanced, chat) + **LLM 원본 출력** |
| 복장 통합 I/O 로그 | `logs/prompt_io.log` | haganai_extract_outfit_prompt의 입력(outfits, chats, previous_result) + 출력(JSON) |
| 일관성 추적 로그 | `logs/mode_operation.log` | prompt_enhance_mode의 `consistency_tracking` 액션: previous_enhanced 전달 여부, 캐시된 캐릭터 목록 |

### 로그로 문제 진단하는 법

1. **세나 복장이 계속 다르게 나옴** → `enhance_prompt_io.log` 확인:
   - `has_previous_enhanced: true` 인지 확인 → `false`면 추적이 안 되고 있음
   - `previous_enhanced` 내용과 LLM 출력 비교 → LLM이 무시하고 다르게 쓰는지 확인
   - `outfit_result`가 배치마다 다른지 확인

2. **VLM 환각 의심** → `prompt_io.log` 확인:
   - outfit_prompt_history vs positive_prompt_history 비교
   - VLM만 보이는 항목(positive에 없는 항목)이 있는지 확인
