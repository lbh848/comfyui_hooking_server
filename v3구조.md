# ComfyUI 후킹 프록시 서버 v3 - 전체 구조 문서

> 컨텍스트 없이도 즉시 작업을 시작할 수 있도록 작성된 구조 가이드

---

## 1. 시스템 개요

```
RisuAI(클라이언트)
    ↓ HTTP/WS (포트 8189)
[후킹 프록시 서버]  ← 이 프로젝트
    ↓ HTTP/WS (포트 8188)
ComfyUI (실제 이미지 생성)
```

- **핵심 역할**: ComfyUI 앞단에 위치하는 프록시 서버. RisuAI 등 클라이언트의 이미지 생성 요청을 가로채서 배치 처리, 프롬프트 강화, 복장 추적 등의 기능을 수행 후 ComfyUI에 전달
- **포트**: 8189 (프록시) → 8188 (ComfyUI)
- **시작**: `python server.py`

## 3. 핵심 데이터 흐름

### 3.1 일반 요청 (배치 모드 OFF)

```
RisuAI → POST /prompt (프롬프트 포함)
  → server.py: handle_prompt()
    → extract_prompts_by_title()  # "긍정프롬프트"/"부정프롬프트" 노드에서 텍스트 추출
    → clamp_weights()             # 가중치 클램프 (선택)
    → process_prompt()
      → build_prompt()            # API 워크플로우에 프롬프트 주입
      → submit_to_real_comfy()    # 8188로 전송
      → wait_for_real_comfy()     # WebSocket으로 완료 대기
      → fetch_real_image()        # 결과 이미지 다운로드
      → save_backup()             # WebP + JSON 백업 저장
    → RisuAI에 이미지 반환 (/view 엔드포인트)
```

### 3.2 배치 모드 (batch_mode_enabled=true)

```
RisuAI → POST /prompt
  → handle_prompt()
    → split_prompt_chat()         # [CHAT] 섹션 분리
    → clamp_weights()             # 가중치 클램프
    → batch_mode.add_request()    # 요청 큐에 추가 → 검은색 이미지 즉시 반환
    → [N초 타임아웃 대기]
    → batch_mode._process_batch()
      → (1) before_generate_func → enhance_mode.enhance_prompt()  # 프롬프트 강화
      → (2) generate_image_func  → generate_image_with_prompt()   # 실제 이미지 생성
      → (3) save_backup_func     → save_backup()                  # 백업 저장
      → (4) on_batch_complete    → outfit_mode.process_batch_images() # 복장 추출
        → 각 이미지를 ComfyUI에 업로드
        → 복장 추출 워크플로우 실행
        → 결과 파싱 (캐릭터별 분리)
        → character_results 업데이트
        → on_processing_complete → _do_restore_workflow()  # 원래 워크플로우 복원

재전송 예약 시:
  다음 요청 → batch_mode.get_scheduled_image() → 캐시된 이미지 즉시 반환
```

## 4. 모드별 상세

### 4.1 BatchMode (`modes/batch_mode.py`)

| 항목 | 내용 |
|------|------|
| **목적** | 연속된 이미지 생성 요청을 모아서 일괄 처리 |
| **동작** | 요청 → 검은 이미지 즉시 반환 → 타임아웃 후 일괄 생성 → 캐시 |
| **키 데이터** | `BatchRequest` (positive, negative, chat_content, image_bytes) |
| **재전송** | `scheduled_batch` - 완성된 배치를 다음 요청 시 매칭하여 반환 |
| **설정** | `batch_mode_enabled`, `batch_timeout_seconds` (config.json) |

**중요**: 배치 모드는 다른 모드들의 부모. `on_batch_complete` 콜백으로 outfit_mode가 트리거되고, `before_generate_func`로 enhance_mode가 실행됨.

### 4.2 OutfitMode (`modes/outfit_mode.py`)

| 항목 | 내용 |
|------|------|
| **목적** | 생성된 이미지에서 캐릭터별 복장을 추출하여 추적 |
| **동작** | 배치 완료 → 각 이미지를 ComfyUI에 업로드 → 복장 추출 워크플로우 실행 → 결과 파싱 |
| **키 데이터** | `character_results: dict[str, CharacterOutfitData]` |
| **파싱 형식** | `캐릭터1=+=복장프롬프트1 \|\| 캐릭터2=+=복장프롬프트2` |
| **영속화** | `workflow_backup/mode/outfit_mode/{캐릭터명}/data.json` + 이미지 |
| **LLM 통합** | `run_llm` API → `customprompt/`의 스크립트 실행 → JSON 결과 저장 |

**CharacterOutfitData 구조**:
```
name: str                          # 캐릭터 이름
entries: list[CharacterOutfitEntry] # 최대 10개 (시간순, [0]=오래됨, [-1]=최신)
  ├── outfit_prompt: str           # VLM이 추출한 복장 설명
  ├── positive_prompt: str         # 이미지 생성에 사용된 프롬프트
  ├── chat_content: str            # 당시 채팅 내용
  └── image_bytes: bytes           # 추출된 캐릭터 이미지
llm_result: Optional[str]          # LLM 복장 통합 결과 (JSON)
llm_dirty: bool                    # 새 엔트리 추가 후 LLM 재실행 필요 여부
```

### 4.3 PromptEnhanceMode (`modes/prompt_enhance_mode.py`)

| 항목 | 내용 |
|------|------|
| **목적** | 이미지 생성 전 프롬프트의 복장 묘사를 일관성 있게 강화 |
| **동작** | 배치 타임아웃 후, 각 요청 처리 전 콜백으로 실행 |
| **입력** | [UPSCALE] 섹션의 `|` 구분 캐릭터 블럭 |
| **참조** | `outfit_mode.character_results`에서 캐릭터별 LLM 결과 가져옴 |
| **출력** | 강화된 프롬프트로 [Positive], [ILXL], [UPSCALE] 섹션 교체 |
| **일관성** | `_last_enhanced_block`으로 이전 강화 결과 추적 |

**프롬프트 구조 (RisuAI에서 전송)**:
```
[Positive]
  ...일반 태그...
[ILXL]
  ...ILXL용 태그...
[UPSCALE]
  캐릭터1 태그들 | 캐릭터2 태그들 | 캐릭터3 태그들
[CHAT]
  채팅 내용...
```

### 4.4 LLM Service (`modes/llm_service.py`)

| 서비스 | 설명 |
|--------|------|
| **copilot** | GitHub Copilot API (`api.githubcopilot.com`) |
| **vertex** | Google Vertex AI SDK |
| **customapi** | 사용자 지정 HTTP 엔드포인트 (OpenAI 호환) |

**공개 함수**:
- `callLLM(messages)` → LLM1 호출 (기본)
- `callLLM2(messages)` → LLM2 호출 (폴백용)
- `update_config(config)` → 서비스/모델 설정 업데이트

**폴백 로직**: customprompt 스크립트에서 LLM1 실패 시 `[LLM 실패]`로 시작하면 LLM2로 재시도.

## 7. customprompt/ 스크립트 인터페이스

모든 스크립트는 `async def run(...)` 함수를 노출해야 함. `importlib`로 동적 로드됨.

### 복장 정리 스크립트 (`outfit_prompt_file`)
```python
async def run(character_name: str, outfit_list: list, chat_list: list = [],
              previous_result: str = None) -> str
# outfit_list: [{"outfit_prompt": "...", "positive_prompt": "..."}, ...]
# 반환: JSON 문자열 {"headgear":"...", "clothing":"...", "shoes":"...", "accessories":"...", "memo":"..."}
```

### 프롬프트 강화 스크립트 (`enhance_prompt_file`)
```python
async def run(character_name: str, char_block: str, outfit_result: str = None,
```

## 8. 놓치기 쉬운 포인트

### 8.1 워크플로우 로딩 & 캐싱
- `update_workflow_if_needed()`는 SHA256 해시로 변경 감지. 해시가 같으면 `current_work/`의 캐시 사용
- ComfyUI `/workflow/convert` 엔드포인트로 원본 → API 포맷 변환
- 복장 추출 워크플로우도 동일한 캐싱 방식 (`current_mode_workflow/`)

### 8.2 프롬프트 주입 방식
- 워크플로우 노드의 **title**이 `"긍정프롬프트"`인 노드의 `inputs.value`에 텍스트 주입
- 프롬프트에 `[Positive]`, `[ILXL]`, `[UPSCALE]`, `[CHAT]` 섹션이 있음

### 8.3 배치 모드 파이프라인 병렬 처리
- `_process_batch()`에서 `enhance_pipeline()`과 `generate_pipeline()`을 `asyncio.gather`로 동시 실행
- 강화 1개 완료 → 즉시 이미지 생성 시작 → 다음 강화 동시 진행

### 8.4 재전송 매칭 로직
- `get_scheduled_image()`는 들어오는 프롬프트를 정규화(가중치 제거)하여 캐시된 프롬프트와 매칭
- 강화 전 원본 프롬프트(`original_processed_positive`)로 매칭 (강화 후는 원본과 다름)

### 8.5 싱글톤 패턴
- 모든 모드는 모듈 레벨 싱글톤: `batch_mode`, `outfit_mode`, `enhance_mode`, `mode_logger`
- `server.py`에서 함수 의존성을 주입 (함수 정의 후 `batch_mode.generate_image_func = ...`)

### 8.6 모드 간 참조 관계
- `enhance_mode.outfit_mode_ref = outfit_mode` → 강화 시 복장 결과 참조
- `batch_mode.on_batch_complete = outfit_mode.process_batch_images` → 배치 완료 시 복장 추출
- `batch_mode.before_generate_func = _before_generate_enhance` → 이미지 생성 전 강화
- `outfit_mode.on_processing_complete = _do_restore_workflow` → 복장 처리 후 원래 워크플로우 복원

### 8.7 워크플로우 복원
- 복장 추출이 끝나면 원래 워크플로우로 전환하여 가중치를 VRAM에 프리로드
- `restore_prompt_file`의 스크립트가 임의 프롬프트를 생성하여 1회 실행

### 8.8 이미지 포맷 변환
- 클라이언트 전송 시 `IMAGE_FORMAT` 설정에 따라 변환 (webp/png/jpeg/original)
- 백업은 항상 WebP (quality=80) + EXIF에 원본 워크플로우 포함

### 8.9 로그 파일들
- `prompt_io.log`: 복장 정리 LLM의 입력/출력 기록
- `enhance_prompt_io.log`: 강화 LLM의 입력/출력 기록
- `mode_operation.log`: 모드 작동 로그 (100KB 초과 시 자동 잘라냄)
- `llm_service.log`: LLM 서비스 호출 로그
